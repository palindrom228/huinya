# suspend под капотом: CPS, Continuation, state machine

## Краткий ответ (TL;DR)

`suspend` функция компилируется в обычный JVM-метод с **дополнительным параметром** `Continuation<T>` (CPS — continuation-passing style). Тело превращается в **state machine**: каждая точка suspension становится `case` в `switch` по `label`. При вызове suspending API внутри — функция сохраняет state и возвращает `COROUTINE_SUSPENDED`. Когда suspension завершается — `Continuation.resumeWith(result)` снова вызывает тот же метод с обновлённым label. Никакой "магии потоков" нет — обычный generated bytecode.

## Развёрнутый ответ

### CPS transform

```kotlin
suspend fun fetchUser(): User = ...

// Компилируется примерно в:
fun fetchUser(continuation: Continuation<User>): Any? { ... }
```

Возврат `Any?`:
- Либо результат `User` (если выполнено синхронно).
- Либо `COROUTINE_SUSPENDED` (если suspending API внутри ушло в suspension).

### State machine

```kotlin
suspend fun loadProfile(id: Long): Profile {
    val user = fetchUser(id)        // suspension point 1
    val avatar = fetchAvatar(user.avatarId)  // suspension point 2
    return Profile(user, avatar)
}
```

Компилируется в нечто похожее на:

```kotlin
fun loadProfile(id: Long, $cont: Continuation<Profile>): Any? {
    val sm = $cont as? LoadProfileSM ?: LoadProfileSM($cont)
    val result = sm.result

    when (sm.label) {
        0 -> {
            sm.id = id
            sm.label = 1
            val r = fetchUser(id, sm)
            if (r === COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
            sm.user = r as User
            // fallthrough
        }
        1 -> {
            sm.user = result as User
            sm.label = 2
            val r = fetchAvatar(sm.user.avatarId, sm)
            if (r === COROUTINE_SUSPENDED) return COROUTINE_SUSPENDED
            return Profile(sm.user, r as Avatar)
        }
        2 -> {
            return Profile(sm.user, result as Avatar)
        }
    }
}

class LoadProfileSM(completion: Continuation<*>) : ContinuationImpl(completion) {
    var label: Int = 0
    var id: Long = 0
    var user: User? = null
    var result: Any? = null

    override fun invokeSuspend(result: Any?): Any? {
        this.result = result
        return loadProfile(id, this)
    }
}
```

Каждая локальная переменная, переживающая suspension, становится полем state-машины.

### Continuation interface

```kotlin
interface Continuation<in T> {
    val context: CoroutineContext
    fun resumeWith(result: Result<T>)
}
```

Когда асинхронная операция завершается — вызывает `continuation.resumeWith(Result.success(value))` или `Result.failure(error)`. Это перезапускает state machine с новым label.

### Почему важно понимать

- **Нет потоков на каждую корутину**: 100 тыс. корутин — это 100 тыс. лёгких state-машин, а не 100 тыс. потоков.
- **Suspension ≠ blocking**: `delay(1000)` не блокирует поток, а планирует resume через 1 сек.
- **Цена suspend**: каждая `suspend` функция = аллокация state-объекта. Inline suspend (`inline suspend fun`) — без аллокации (compiler разворачивает).
- **Stack traces**: видно меньше деталей, потому что фреймы не настоящие. Используй `DEBUG_PROPERTY_NAME = "on"` или `kotlinx-coroutines-debug` для лучших traces.

### suspendCoroutine / suspendCancellableCoroutine

Низкоуровневая точка интеграции с callback-API:

```kotlin
suspend fun fetchUserCallback(id: Long): User = suspendCancellableCoroutine { cont ->
    api.fetchUser(id, object : Callback {
        override fun onSuccess(user: User) = cont.resume(user)
        override fun onError(e: Throwable) = cont.resumeWithException(e)
    })
    cont.invokeOnCancellation { api.cancel(id) }
}
```

`suspendCancellableCoroutine` — поддерживает cancellation (всегда предпочтительнее).

### Inline suspend

```kotlin
inline suspend fun <T> measure(block: () -> T): T {
    val start = System.nanoTime()
    val result = block()
    val elapsed = System.nanoTime() - start
    log(elapsed)
    return result
}
```

Inline разворачивается в месте вызова → нет state-машины, нет аллокации Continuation, **но** `block` не может содержать suspend (если только не помечен `crossinline`/`noinline` правильно).

### context preservation

Continuation хранит CoroutineContext (Job, Dispatcher, etc.). Когда `resumeWith` вызывается — следующий шаг state-машины выполняется на нужном dispatcher'е (через `intercepted()`).

### Cancellation в state machine

```kotlin
// Сгенерированный код проверяет на каждом suspension point:
if (!isActive) throw CancellationException()
```

Поэтому код без suspension points не cooperative — нужны явные `yield()` / `ensureActive()`.

## Подводные камни

- **Suspension point — точка отмены**: между ними cancellation не сработает. CPU-bound loop без yield → не отменяется.
- **Stack trace неполный** — нужен debug mode coroutines.
- **`runBlocking`** — создаёт цикл event loop в текущем потоке. Не использовать на Main thread в production.
- **suspending function ≠ async** — она просто маркер, что внутри **может быть** suspension. Если внутри только синхронный код — выполнится синхронно без аллокаций state-машины (некоторые случаи).
- **invokeSuspend выбрасывает CancellationException** — нужно правильно ловить (только если хочешь cleanup, re-throw обязательно).
